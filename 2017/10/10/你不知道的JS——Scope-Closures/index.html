Scope&Closures

Chapter1:What is Scope？

编译器理论

传统语言的编译过程：

  1. Tokenizing／Lexing

    将代码分割成有意义的块，这些块就是所谓的tokens。

  2. Parsing

    这个阶段将tokens转化成AST树（抽象语法树）

  3. Code-Generation

    将ast树转化为可执行代码

JavaScript 引擎不像其他语言在编译上花费很多时间，所以为了更好的性能。js引擎使用很多技巧（比如JITs，lazy compile 甚至hot re-compile），所以说js语句在执行前会编译。

理解作用域

角色

  1. Engine
  2. Compiler
  3. Scope：收集和维护了一张所有声明标识（变量）的查找表，并且制定了这些标识能否访问的规则

Back&Forth

我们来看看引擎和上面的这些角色是如何解释var a =2这个程序的。

Compiler：

1、当看到var a，Compiler询问Scope是否变量a已经存在于特定的作用域集合（scope collection）了。如果存在，那么Compiler就会忽略这个声明并继续处理，如果不存在，Compiler则会要求Scope声明一个新的变量在当前的作用域集合中。

2、接下来，Compiler生成出供Engine执行的语句（a=2），Engine会先查找Scope是否有一个变量a在当前的作用域集合，如果有，就进行赋值操作，如果没有，Engine则去其他地方寻找。

如果最终Engine找到了变量，那么将2赋值进去，如果没有，就抛出错误。

Compiler Speak

LHS “Left-hand Side”:赋值操作，(who’s the target of the assignment)

RHS “Right-hand Side”:非LHS，例如console.log(a),(who’s the source of the assignment)

你说对了，还是它，没变。给你。

Engine: Cool. Passing the value of a, which is 2, into log(..).

酷，那我就把a的值2传入log吧。

Nested Scope 嵌套作用域

我们说Scope是一些通过变量的名字来寻找它们的规则，但通常不会只有一个Scope。

如果一个变量不能在当前的作用域找到的话，那么就会通过作用域链往上寻找直到找到全局而停止。

Errors

为什么要有LHS和RHS的区分呢，因为这两种方式在找不到变量的时候会有区别。

当RHS找不到一个变量的时候，结果是ReferenceError类型的错误被Engine抛出。

相反，如果LHS查找到作用域顶层（global Scope）没有找到，那么如果不是在严格模式下，全局作用域会创建这个变量。

如果变量在RHS查找时存在，但是对这个变量做的事是不可能的，如变量值为null或undefined。Engine会抛出TypeError的错误类型。

Chapter 4:Hoisting（提升）

The Comiler Strikes Again

所有的声明，包括变量和函数的声明，都会先进行处理。

当你看到var a =2，你可能仅仅把它看作是一条语句。但是js却将它看成两条语句：var a;anda=2.第一条语句在编译阶段（compilation phase）处理，第二条语句则在执行阶段（execution phase）适时地处理。

注意：只有声明才提升。赋值过程不提升。

这就造成了函数声明会提升，而函数表达式并不提升（只提升了声明变量）。

Functions First

函数先提升，然后是变量。
